## 0. Как GPT 5.5 должен работать с этим планом

### 0.1 Протокол взятия задачи

1. Берёшь **первую не-`[x]` задачу** в порядке из §1.3 (порядок исполнения), если пользователь не сказал иное.
2. Читаешь только файлы из блока «Файлы» этой задачи + контракты из шапки. Не сканируй весь репозиторий.
3. Отмечаешь соответствующий item в `TASK_QUEUE.md` как `[~]`, по завершении `[x]` с датой и списком пройденных гейтов.
4. Выполняешь микрошаги задачи **строго по порядку** (P1.2.3 не начинать, пока P1.2.2 не зелёный).
5. Каждый микрошаг рассчитан на 15–60 минут работы. Если微 шаг разрастается — останавливаешься, фиксируешь сделанное, дробишь остаток на новые микрошаги в TASK_QUEUE (не в этом файле).
6. После задачи прогоняешь её гейт-профиль (§0.3) целиком, и только потом переходишь к следующей.
7. Коммиты в git — **только по явной просьбе пользователя**.

### 0.2 Правило наследования полей

Полный набор полей (приоритет, цель, почему, файлы, acceptance, команды, риски, зависимости, expected diff type, rollback) задаётся на уровне **задачи** (P1.2). **Микрошаги** (P1.2.3) наследуют поля родителя; у каждого микрошага явно указаны: действие → выход → критерий готовности. Если у микрошага есть собственный риск/гейт — он указан прямо в строке микрошага.

### 0.3 Гейт-профили (используются в задачах по ссылке)

| Профиль | Команды | Когда |
|---|---|---|
| **G0** | `npm run check` | любые правки TS/Svelte |

Базовые числа на сегодня: vitest ≈ 306 тестов, e2e ≈ 121 тест, бюджеты бандла проверяет `scripts/check-bundle-budget.mjs`.

### 0.4 Expected diff type — словарь

- `code-new-module` — новый файл(ы) в `src/lib/**`, существующие почти не трогаются.
- `code-edit` — точечные правки существующих файлов.
- `code-move` — перенос кода без изменения поведения (распил монолита).
- `schema-migration` — bump `SCHEMA_VERSION` + миграционный блок + тесты (только по `docs/SCHEMA_MIGRATION_PROCESS.md`).
- `test-only` — только тесты/фикстуры/снапшоты.
- `infra` — `scripts/*`, `mcp/*`, CI, деплой; бандл приложения не трогается.
- `ops` — действия вне репозитория (Cloudflare, Supabase, домен) + фиксация в доках.
- `docs` — только markdown.

### 0.5 Жёсткие инварианты (нарушение = брак задачи)

1. Все мутации состояния — через канонические пути `updateElement`/`updateOrphan`/хелперы `elementTree.ts` (см. `MUTATION_UPDATE_RULES.md`, `AGENT_AUTONOMY/MUTATION_PATH_CONTRACT.md`).
2. Одна пользовательская/агентская операция = ровно одна запись undo-истории; no-op не пишет историю (паттерн item 181).
3. Любое изменение `ProjectPayload`/`StudioState` = bump схемы + миграция + тесты preserve/normalize + roundtrip envelope/import.
4. Никаких секретов в клиенте/репо; base64 не пишется в `state_json`.
5. URL/HTML/SVG от пользователя или агента — только через `src/lib/security/urls.ts`, `src/lib/export/htmlSanitizers.ts`, `src/lib/security/svgSanitizer.ts`.
6. Новые UI-строки — синхронно добавлять в `src/lib/i18n/uiRuntimeLocale.ts` (RU-локаль матчится по текстам!).
7. Тяжёлые зависимости в бандл приложения не добавлять; MCP SDK живёт в `mcp/`, не в `src/`.
8. Electron (`electron/`) — заморожен, не трогать.

# P1 — Clean semantic HTML export / responsive / no lock-in

Архитектурное решение (из roadmap §5.2): вводим **export layout mode** — `flow` (новый дефолт для новых проектов: вертикальный поток, ряды → горизонтальный flex, размеры → max-width/проценты) и `absolute` (legacy, текущее поведение, дефолт миграции для старых проектов). Режим хранится в `ProjectExportSettings` + переопределение на фрейме; элемент получает флаг `exportPinned?: boolean` («экспортировать абсолютом поверх потока»).

## P1.1 Схема v23: `exportLayoutMode` + `exportPinned`

- **Приоритет:** критический.
